Le process de la Step 1
Read the Market

Ce que fait la Step 1, étape par étape — de l'input brut au rapport de marché publié. Cette page documente la version de la skill en vigueur au 12 juin 2026 ; sa définition exacte est téléchargeable en bas de page.

Ce qu'un run produit

Une exécution de la skill = un run : un rapport figé et daté, publié sous market.shoette.com/[marché]/s1-[N]_[date]/, accompagné de son data.json — le canevas de données structurées qui alimente le graphique interactif et servira aux skills suivantes. À chaque run, la page du marché et l'accueil sont mis à jour.

Le rapport contient quatre blocs :

  • Executive Summary — le cadrage du marché en une demi-page : les dimensions qui le structurent (la grille de lecture de tout le reste), et les hypothèses prises sur l'input.
  • 2a — Part de marché & croissance — le nuage de points part × croissance, précédé du stade de maturité et du choix justifié de la période de référence.
  • 2b — Positionnement interactif — le graphique à 5 canaux visuels (calques, fiches acteurs, export annoté), suivi du tableau de toutes les valeurs et du bloc « Comment lire ces valeurs » (unités, devises, conversions).
  • Annexes — A1 : toutes les sources (URL + date de consultation) ; A2 : le lexique du secteur, défini en français et en anglais.

Le déroulé, étape par étape

Étape 0 — Résoudre le contexte du run

La skill identifie le marché (slug ASCII, réutilisé si le marché existe déjà), calcule le numéro du run (dernier run S1 + 1), fixe la date du jour — d'où le dossier s1-[N]_[date] — et adopte la langue de l'input pour tout le rapport (les pages parent du site restent en français). Un contrôle préalable vérifie que le moteur du graphique interactif existe : sans lui, la skill refuse de démarrer.

Étape 1 — Interpréter l'input

L'input peut être un secteur, un besoin exprimé en langage naturel, une zone géographique — ou n'importe quelle combinaison. S'il est incomplet, la skill infère le marché le plus probable et affiche ses hypothèses en tête de rapport plutôt que de demander. Elle ne pose une question de clarification que si l'input est trop ambigu pour avancer.

Étape 2 — Rechercher

Des sous-agents de recherche partent en parallèle sur le web (presse économique internationale et presse spécialisée du secteur) — ils tournent sur un modèle économique (Sonnet), la consolidation restant au meilleur modèle de la session. Ils établissent : le stade de maturité du marché, qui détermine la période de référence (6 mois, 1 an ou 3 ans, choix justifié) ; les dimensions qui structurent le marché, classées par importance ; et pour chaque acteur significatif, la part de marché estimée, la croissance et une valeur par dimension. Trois règles absolues : toute estimation est signalée, aucun acteur significatif n'est omis en silence (faute de données chiffrées, il est listé à part avec ce qui a été cherché), et les données brutes ne sont pas conservées — seules survivent les données structurées et les références des sources.

Étape 3 — Construire le canevas de données

Tout est consolidé dans un data.json au schéma fixe : le classement des dimensions y pilote la configuration par défaut du graphique interactif (rang 2 → position X + couleur, rang 3 → taille, etc.) ; chaque valeur porte son drapeau « estimé » ; les identifiants des acteurs et des dimensions sont des slugs stables, conçus pour être appariés d'un run à l'autre par les skills suivantes ; et la note d'export (clé de lecture + première observation du marché) y est rédigée pour être embarquée dans les exports PNG/SVG du graphique.

Étape 4 — Générer la page du run

La page est autonome et figée : les données y sont inlinées et écrites dans le data.json voisin — strictement identiques. Le graphique 2a est un SVG statique généré pour ce run ; le 2b est monté sur le moteur partagé et versionné du site (construit une fois, jamais reconstruit pendant un run) ; sous le graphique, le tableau de toutes les valeurs puis les notes de lecture — unité et devise utilisées, conversions éventuelles avec taux de change, signification des échelles. La page se termine par les annexes et le champ « dernière mise à jour », seul élément qu'une future skill aura le droit de modifier.

Étape 5 — Mettre à jour les pages parent

La page du marché (section Step 1) et l'accueil sont mis à jour uniquement entre leurs balises RUNS-SKILL1 et MARKETS — les runs du plus ancien au plus récent, Run 1 en premier. Tout le reste de ces pages appartient à Elena et n'est jamais écrasé. Au premier run d'un marché, la page marché est créée depuis le template du site.

Étape 6 — Déployer, vérifier, committer

Déploiement en deux temps : une simulation (--dry-run) pour contrôler la liste des fichiers, puis l'envoi réel — vers le répertoire market/ de l'hébergement exclusivement, un garde-fou du script refusant tout autre chemin. La skill vérifie ensuite que les pages répondent en ligne, puis committe le run dans Git : chaque run est versionné, l'historique est rejouable.

Les garde-fous

  • Estimations toujours signalées — le préfixe « ~ » suit chaque valeur estimée, du data.json jusqu'aux exports.
  • Jamais d'omission silencieuse — un acteur sans données chiffrées reste visible, avec la trace de ce qui a été cherché.
  • Runs figés — un run publié n'est jamais modifié (hors champ « dernière mise à jour », réservé à une skill future).
  • Périmètre distant strict — la skill n'écrit que dans le répertoire market/ de l'hébergement.
  • Modèles répartis — recherche sur Sonnet, orchestration et rédaction sur le meilleur modèle disponible.
  • Traçabilité — sources datées en annexe, données structurées versionnées par Git, données brutes jetées.
